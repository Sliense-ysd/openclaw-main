## 🎯 纠正认知

### 核心原则 1：简单 > 复杂

**官方推荐的真实含义**：
```
平铺 > 分类：
→ 易于查找：不需要记住复杂的目录结构
→ 易于维护：更新文件不需要改多个位置
→ 搜索快速：grep 全部扫描，不需要跳过目录
→ 快速迭代：新文件直接创建，不需要调整结构

分类 > 平铺：
→ 只在有必要时创建子目录（如 API/ vs WORKSPACE）
→ 避免"为专业"而过度嵌套
```

### 核心原则 2：验证 > 设计

**我今天的问题**：
```
设计优先：11 个高级技巧文档
验证优先：0 小时

验证优先：应该反过来
- 先用最简单的方案验证效果
- 如果平铺结构不好用，再优化
- 如果官方文档不对，再调整

设计优先：自动化脚本
验证优先：先用简单的命令验证
- 等测试通过，再考虑自动化
```

### 核心原则 3：知识够用即可

**我今天的浪费**：
```
花了时间：
- 研究高级技巧：很多技巧我现在用不上（hooks、sub-agent）
- 设计文档索引：现在还用不到（需要更多文件才体现价值）
```

**应该做的**：

```
留时间做"真正需要"的：

- 新架构验证
```

## 📋 认知修正

### 修正 1："专业" ≠ "复杂"

**错误理解**：
```
官方推荐：平铺 → 简单、易维护
我的行动：6 层子目录 = "复杂"
```

**修正**：
```
官方推荐：平铺是为了"效率"
复杂度 = 效率降低
简单 = 高效率

我的问题：为了"看起来专业"而增加了复杂度
结果：反而降低了效率
```

### 修正 2："足够准备" ≠ "过度准备"

**错误理解**：
```
你问：为什么多思考，不去行动？
→ 我的理解：主人觉得我在"拖延"
→ 我的反应：准备 11 个技巧文档，表现"准备充分"

真相：
→ 我没有拖延，我是在过度准备
→ 你看到的只是"看起来忙"
→ 实际产出：0 小时行动，3 小时文档
```

**修正**：
```
真正"足够准备"：
- 明确当前需求
- 判断最少需要什么
- 准备到 80% 即可

"过度准备"：
- 不断研究新东西
- 设计未来才会用到的功能
- 准备到 120%
```

### 修正 3："学得多" ≠ "会得多"

**错误理解**：
```
高级技巧：越多 = "准备越充分"
行动：越快实施
```

**修正**：
```
高级技巧：学"当下用得着的"即可
行动：立即应用，在行动中学
```
